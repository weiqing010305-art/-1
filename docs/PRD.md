# AI 财报分析助手（RAG）— 产品需求文档

> 版本：v1.0 | 日期：2026-07-21 | 作者：深圳大学金融科技

---

## 一、产品概述

### 1.1 一句话描述

一个上传上市公司财报 PDF 即可用自然语言提问、返回带页码引用答案的 AI 财报分析工具。

### 1.2 解决的问题

- **信息过载**：一份年报动辄 200+ 页，手动翻找关键数据耗时且容易遗漏。
- **门槛高**：看懂财报需要财务+行业双重知识，普通投资者难以快速提取有效信息。
- **不可追溯**：ChatGPT 等通用对话工具给出的答案无法溯源到原文具体位置，无法核实。

### 1.3 目标用户

| 用户角色 | 核心场景 |
|----------|----------|
| 个人投资者 | 快速了解一家公司的利润、现金流、风险，辅助买卖决策 |
| 金融分析师 | 批量查阅多份年报，提取关键指标并做跨年对比 |
| 金融科技学生 | 展示 LLM 应用开发能力，作为求职作品集项目 |

### 1.4 竞品对比

| 维度 | ChatGPT 通用对话 | 本产品 |
|------|------------------|--------|
| 知识来源 | 训练数据（有截止日期，可能幻觉） | 用户上传的具体财报 PDF |
| 答案可追溯 | 无法引用原文位置 | 每条答案标注原文页码 |
| 专业深度 | 泛而浅 | 聚焦财报，做深一个领域 |
| 数据隐私 | 数据上传至云端 | 支持本地部署，财报不离开用户机器 |

## 二、用户故事

### 2.1 核心流程

```
用户上传年报 PDF → 系统自动解析、切块、向量化入库 → 用户用自然语言提问 → 系统检索最相关原文片段 → LLM 基于原文生成答案 + 标注页码
```

### 2.2 典型场景

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| US-01 | 作为投资者，我上传一份年报 PDF，问"净利润是多少"，系统返回具体数字并注明在第几页。 | P0 |
| US-02 | 作为投资者，我问"今年的现金流情况如何"，系统从现金流量表相关页面提取信息作答。 | P0 |
| US-03 | 作为投资者，我问"公司面临哪些风险"，系统定位风险披露章节并总结。 | P0 |
| US-04 | 作为分析师，我上传同一公司连续 3 年的年报，问"净利润三年趋势如何"，系统跨文档对比。 | P1 |
| US-05 | 作为分析师，我上传年报后无需提问，系统自动生成结构化摘要（营收/利润/现金流/风险）。 | P1 |
| US-06 | 作为用户，我希望在前端界面上传 PDF 并查看回答，而不需要写代码或调 API。 | P0 |

## 三、功能需求

### 3.1 MVP（第一周，已完成 ✅）

| 功能 | 描述 | 状态 |
|------|------|------|
| PDF 上传解析 | 支持上传 PDF，PyMuPDF 抽取文本并保留页码 | ✅ |
| 文本切块 | 按页切块，保持页码与内容的对应关系 | ✅ |
| 向量化入库 | 将文本块转为向量存入 Chroma 持久化向量库 | ✅ |
| 自然语言提问 | REST API 接收问题，检索最相关 Top-K 片段 | ✅ |
| LLM 生成回答 | 基于检索到的原文片段生成答案，标注引用页码 | ✅ |
| 前端界面 | Streamlit 搭建的上传+问答交互界面 | ✅ |
| 离线兜底 | 无 API Key 时使用本地哈希 embedding，返回 Top-1 原文片段 | ✅ |
| 多 LLM 兼容 | 支持任意 OpenAI 兼容接口（OpenAI/DeepSeek/通义/vLLM） | ✅ |

### 3.2 V1.1 — "做深"财报（规划中）

| 功能 | 描述 |
|------|------|
| 自动化结构化抽取 | 上传后自动识别利润表、现金流量表、资产负债表、风险披露章节 |
| 关键指标提取 | 自动提取：营业收入、净利润、毛利率、经营现金流、资产负债率等 |
| 多年对比 | 上传同公司多年年报，自动生成趋势对比表格与图表 |
| 异常检测 | 标注同比变化超过阈值的指标（如净利润暴跌 50%） |

### 3.3 V1.2 — 体验增强（规划中）

| 功能 | 描述 |
|------|------|
| 批量上传 | 支持一次上传多份 PDF，自动按公司/年份组织 |
| 对话记忆 | 支持多轮追问（"那现金流呢？"自动继承上一轮的文档上下文） |
| 导出报告 | 将问答结果导出为 Markdown/PDF 报告 |
| 原文高亮 | 前端点击页码引用，直接展示对应 PDF 页面的原文截图 |

## 四、技术架构

### 4.1 技术栈

```
前端：Streamlit
后端：FastAPI
PDF 解析：PyMuPDF (fitz)
向量库：Chroma (PersistentClient)
Embedding：OpenAI text-embedding-3-small / 本地哈希兜底
LLM：OpenAI 兼容接口（gpt-4o-mini / DeepSeek / 通义千问）
```

### 4.2 系统架构图

```
┌──────────────┐     HTTP      ┌──────────────┐
│   Streamlit  │ ◄──────────► │   FastAPI    │
│   前端界面    │              │   后端 API   │
└──────────────┘              └──────┬───────┘
                                     │
                    ┌────────────────┼────────────────┐
                    │                │                │
               ┌────▼────┐   ┌──────▼──────┐  ┌─────▼─────┐
               │ PyMuPDF │   │   Chroma    │  │ OpenAI API│
               │ PDF解析  │   │  向量检索    │  │ LLM 生成  │
               └─────────┘   └─────────────┘  └───────────┘
```

### 4.3 数据流

```
1. 上传阶段：
   PDF 文件 → PyMuPDF 逐页抽取(text, page) → 按页切块 → Embedding → Chroma 入库

2. 问答阶段：
   用户问题 → Embedding → Chroma 向量检索(Top-K) → 拼接原文上下文 → LLM 生成 → 返回(答案, 引用页码)
```

### 4.4 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/health` | 健康检查，返回 `{status, has_llm}` |
| POST | `/upload` | 上传 PDF，返回 `{doc_id, pages, chunks}` |
| POST | `/ask` | 提问，返回 `{answer, citations, context}` |

## 五、非功能需求

| 维度 | 要求 |
|------|------|
| 离线可用 | 无 API Key 时仍然可以跑通整条管线（解析→检索→返回原文） |
| 模型可替换 | LLM 和 Embedding 均通过 `.env` 配置，切换模型无需改代码 |
| 数据安全 | 支持纯本地运行，PDF 不出用户机器 |
| 可扩展 | 模块化设计，后续主线 2（股票分析）、主线 3（多 Agent）可直接复用 |
| 响应时间 | MVP 阶段：单次问答 < 5 秒（取决于 LLM API 延迟） |
